# 蒸留ガイド（distill）

このファイルは `#skill-distill` で明示的に参照された時のみ読み込まれます。進化のサイクルの「蒸留」を半自動化する手順を定義します。`.steering/` の振り返りから知見を取り出し、platform-harness（ハーネスのコアリポジトリ）への環流候補を抽出します。**環流の最終判断（PR 作成）は人間が行う。**

実行タイミングの目安: 複数の作業（PR）がマージされたプロジェクトの節目 / platform-harness の更新前の棚卸し / ユーザーの明示的な指示。

## 処理手順

### 1. 処理済み一覧の取得

1. `.steering/` 直下の `distill-*.md`（過去の蒸留レポート）を列挙する
2. 最新レポートの「処理済みディレクトリ」セクションを読み、処理済み一覧とする
3. 過去レポートがなければ処理済みなし（全ディレクトリが対象）

### 2. 振り返りの収集

1. `.steering/` 直下の `YYYYMMDD-` 接頭辞ディレクトリ（パターン: `[0-9]*-*`）の `tasklist.md` を列挙する（`_template/` や `example/` 等の規約外ディレクトリは含めない）
2. 処理済みを除いた各 tasklist.md の「実装後の振り返り」セクションを読む
3. 振り返りが未記入（プレースホルダー `{...}` のまま、または tasklist.md 不在）のディレクトリは「振り返り未記入」として記録し、**処理済みに含めない**（記入後の再実行で拾うため）

### 3. 知見の抽出

「学んだこと」「次回への改善提案」「計画と実績の差分」から、独立した知見を1件ずつ抽出する。1つの箇条書きに複数の知見が含まれる場合は分割する。

### 4. 環流3基準による分類

各知見に以下を順に適用する:

| 基準 | 問い |
|------|------|
| 普遍性 | どのプロジェクトでも有効か（特定の技術スタック・クラウドに依存しないか） |
| 再現性 | 同じ失敗・同じ改善が他のプロジェクトでも起こりうるか |
| 抽象性 | プロジェクト固有の値（URL・モデルID・リージョン等）を含まないか |

- **3つすべて Yes** → 環流候補（platform-harness へ）
- **普遍性のみ No** → 技術スタック層候補（`.kiro/steering/tech.md` へ）
- **それ以外** → プロジェクト内知見（`.kiro/steering/product.md` 等への記録候補）

判定に迷う場合は保守的に「プロジェクト内知見」に分類し、判断根拠を備考に残す。

### 5. レポート出力

`.steering/distill-[今日YYYYMMDD].md` に以下の形式で書き出す。同日のレポートが既存の場合は内容を統合して上書きする。

```markdown
# 蒸留レポート YYYY-MM-DD

## 環流候補（platform-harnessへ）

### 候補1: [タイトル]
- 出典: .steering/YYYYMMDD-xxx/tasklist.md
- 分類根拠: 普遍性Yes / 再現性Yes / 抽象性Yes（[一言の理由]）
- 変更対象ファイル: [platform-harness内の相対パス]
- 変更案: [そのまま使える具体的な変更内容の下書き]
- 参照チェック: [移植物が環流先に存在しない概念・スキル・ツールを参照していないか。参照している場合は一般表現への置換案を記す]

## 技術スタック層候補（.kiro/steering/tech.mdへ）

- [知見と反映案]（出典）

## プロジェクト内知見（.kiro/steering/product.md等へ）

- [知見]（出典） — 記録推奨: [Yes/No と理由]

## 振り返り未記入

- .steering/YYYYMMDD-xxx/（[状態: 未記入 / tasklist不在]）

## 処理済みディレクトリ

- .steering/YYYYMMDD-xxx/
（過去レポートからの引き継ぎ分も再掲し、常に全量を保持する）
```

**変更対象ファイルのパス**は platform-harness のリポジトリ構造（Claude Code 版ハーネス: `.claude/`・CLAUDE.md 汎用層）を前提に書く。環流先は Claude Code ハーネスであるため、Kiro 固有の機構名（`.kiro/steering/` 等）をそのまま書かず、環流先の対応物に読み替える。

### 6. ユーザーへの報告

1. 環流候補の件数と要点を報告し、platform-harness への PR 作成の判断を委ねる
2. プロジェクト内知見のうちセッション横断で有用なものは、`.kiro/steering/product.md` 等の常時参照ファイルへの記録を提案する（Kiro にはセッション横断の永続メモリ機構がないため、steering ファイルが記憶層の代替となる）
3. 振り返り未記入があれば、記入後の再実行を促す

**環流 PR を作成する場合の注記**: 複数 PR の相互参照（PR 番号）は、全 PR を作成した後にまとめて記入・修正する（番号の事前予測は外れる）。PR 間で変更ファイルを重複させず、任意順でマージ可能にする（スタック PR はマージ順事故の温床のため避ける）。
