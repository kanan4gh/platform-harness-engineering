# ドキュメント記入ガイド

このファイルは `#skill-doc-writing` で明示的に参照された時のみ読み込まれます。`docs/` 配下の各永続ドキュメントの記入ガイドを定義します。

## 共通原則

- 具体性と測定可能性: すべての要件を明確・数値で表現する
- ユーザー中心: 機能はユーザーの課題解決に紐づける
- 他ドキュメントとの整合性: 記入後に矛盾がないか確認する

---

## product-requirements.md（プロダクト要求定義書）

**必須セクション**:

| セクション | 内容 |
|---|---|
| プロダクト概要 | 名称・コンセプト・ビジョン・目的 |
| ターゲットユーザー | ペルソナ（基本属性・課題・期待する解決策） |
| 成功指標（KPI） | SMART 原則に基づく定量目標 |
| 機能要件 | ユーザーストーリー + 受け入れ条件（チェックリスト）+ 優先度（P0/P1/P2） |
| 非機能要件 | パフォーマンス・ユーザビリティ・信頼性 |

**優先度の定義**:
- P0（必須）: これがないとプロダクトとして成立しない
- P1（重要）: 初期リリース後すぐに追加すべき
- P2（できれば）: 将来的に検討

## functional-design.md（機能設計書）

**必須セクション**:

| セクション | 内容 |
|---|---|
| 設計指針 | 実装上の判断基準（3〜5項目） |
| システム構成図 | コンポーネント間の関係（テキスト図で可） |
| Claude CLI ↔ Kiro 概念マッピング | 両者の対応表（移行経験者向け） |
| コンポーネント設計 | 各層の詳細設計 |
| ワークフロー設計 | 主要なユースケースのフロー |

## architecture.md（技術仕様書）

**必須セクション**:

| セクション | 内容 |
|---|---|
| アーキテクチャ概要 | 層構造・情報フロー |
| 各層の設計 | 役割・設計方針・ファイル仕様 |
| 環境設定戦略 | devcontainer 非依存の代替手段 |
| 技術的制約 | バージョン依存・既知の制限 |
| ブートストラップ戦略 | フェーズ定義・移行条件 |
| テスト戦略 | 受け入れテスト・ドキュメント品質チェック |

**必須セクション**:

| セクション | 内容 |
|---|---|

**必須セクション**:

| セクション | 内容 |
|---|---|

## glossary.md（ユビキタス言語定義）

**記入形式**:

```markdown
## [用語]

**定義**: [1〜2文の定義]

**使用例**: [文脈での使い方]

**関連用語**: [他の用語へのリンク]
```

**収録対象**:
- プロジェクト固有の用語
- 一般用語だが本プロジェクトで特定の意味を持つもの
- チームメンバー間で定義が曖昧になりがちな用語
