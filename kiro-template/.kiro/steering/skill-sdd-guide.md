# SDD ガイド（スペック駆動開発の詳細手順）

このファイルは `#skill-sdd-guide` で明示的に参照された時のみ読み込まれます。tasklist.md の進捗管理と振り返りの詳細手順を定義します。

## tasklist.md 管理ルール

### タスク実行中の必須動作

- タスク完了時に `[ ]` → `[x]` へ**即座に**更新する
- **完了マークは操作の実行後に付ける**（実行予定の段階で付けない。コミット・PR 作成など、マークの後に実行しがちな操作は特に注意。マークと実態の乖離は後から検出できない）
- 複数タスクをまとめて更新しない（リアルタイムに1件ずつ更新する）
- tasklist.md を確認せずに実装を進めない

### 🚨 タスク完全完了の原則

**全タスクが `[x]` になるまで作業を継続すること。**

以下の理由によるスキップは**絶対禁止**:
- 「時間の都合により後回し」
- 「難しいから次のスプリントで」
- 「テストが面倒」
- 「別タスクとして実施予定」

### タスクが大きすぎる場合

スキップではなく**分割**する:
1. 元のタスクを削除し、その場所にサブタスク（`[ ]` 付き）を追加
2. サブタスクを1つずつ完了させる

### 技術的理由によるスキップ（唯一の例外）

以下の条件をすべて満たす場合のみスキップ可能:
- 実装方針の変更・アーキテクチャ変更・依存関係の変更など**明確な技術的理由**がある
- その理由を tasklist.md に明記する

スキップ時の記述形式:
```
- [x] ~~タスク名~~（実装方針変更により不要: [具体的な技術的理由]）
```

## フェーズ完了時の確認手順

各フェーズ完了後、振り返りを書く前に必ず実行:

1. tasklist.md を読み込む
2. 未完了タスク（`[ ]`）が残っていないか確認する
3. 未完了タスクが見つかった場合:
   - **実装する**（パターンA）
   - **分割して実装する**（パターンB）
   - **技術的理由がある場合のみスキップ**（パターンC）
4. 全タスクが `[x]` を確認してから振り返りへ進む

## 振り返りの記入内容

tasklist.md の「実装後の振り返り」セクションに以下を記入:

| 項目 | 記入内容 |
|---|---|
| 実装完了日 | `YYYY-MM-DD` |
| 計画と実績の差分 | 計画と異なった点・新たに必要になったタスク |
| 学んだこと | 技術的な学び・プロセス上の気づき |
| 次回への改善提案 | 具体的なアクション |
| リリース判断 | 価値のまとまりか・重大バグの有無・バージョン種別 |

## スペック作成の手順

新機能のスペックを作成する際の順序:

1. `.steering/_template/` をコピーして `YYYYMMDD-[機能名]/` を作成
2. `requirements.md` を記入 → ユーザーの確認を得る
3. `design.md` を記入 → ユーザーの確認を得る
4. `tasklist.md` を記入（フェーズ分け・サブタスク明記）→ ユーザーの確認を得る
5. 実装開始
