# 設計書

## アーキテクチャ概要

4つの steering ファイルをそれぞれ独立したマークダウンファイルとして実装する。各ファイルは Kiro の `inclusion: always` フロントマターを持ち、Kiro エージェントの全会話コンテキストに常時注入される。

```
kiro-template/
└── .kiro/
    └── steering/
        ├── process.md    # 開発プロセス原則（ハーネス共通・記入不要）
        ├── product.md    # プロダクト定義（プロジェクト開始時に記入必須）
        ├── tech.md       # 技術スタック・環境設定（プロジェクト開始時に記入必須）
        └── structure.md  # リポジトリ構造要約（プロジェクト開始時に記入必須）
```

## コンポーネント設計

### 1. process.md

**責務**:
- SDD（スペック駆動開発）の基本フローを定義する
- `.steering/YYYYMMDD-xxx/` の使い方を定義する
- kiro-cli 不使用・built-in specs 不使用の方針を明記する

**実装の要点**:
- このファイルはハーネス共通なので、プレースホルダーや `<!-- 例: ... -->` は含めない
- `docs/functional-design.md` の設計指針1〜3を steering ファイルとして要約する
- Claude CLI の `CLAUDE.md` における「汎用層（スペック駆動開発の基本原則）」に相当するが、Kiro 向けに簡潔化する
- ファイル自体がハーネスの「ルールブック」として機能する

**フォーマット**:
```markdown
---
inclusion: always
---

# 開発プロセス原則

## 基本方針
[kiro-cli 不使用・built-in specs 不使用の明記]

## スペック駆動開発フロー
[フロー定義]

## 作業スペックの使い方
[.steering/YYYYMMDD-xxx/ の使い方]
```

### 2. product.md

**責務**:
- プロダクト固有の情報（名称・ビジョン・目的）を Kiro に伝える
- `docs/` 配下のドキュメント一覧を提示し、Kiro が参照先を知れるようにする

**実装の要点**:
- `<!-- 例: ... -->` 形式の記入ガイドを各セクションに含める
- `[プロダクト名を記入]` 形式のプレースホルダーで記入箇所を明示する
- `docs/` の各ファイルへのリンクをプレースホルダーとして含める

**フォーマット**:
```markdown
---
inclusion: always
---

# プロダクト定義

## プロダクト名
<!-- 例: MyApp - ユーザー向けタスク管理サービス -->
[プロダクト名とサブタイトルを記入]

## ビジョン
[ビジョンを記入]

## 目的
[目的を記入]

## 参照ドキュメント
- [product-requirements.md] - PRD
- [functional-design.md] - 機能設計書
...
```

### 3. tech.md

**責務**:
- 技術スタックを Kiro に伝え、適切なコード生成・提案を促す
- OS 別セットアップ手順のプレースホルダーを提供する（devcontainer の代替）
- バージョン固定ファイルの存在を伝える

**実装の要点**:
- devcontainer を使わない前提で、OS 別（macOS・Windows・Linux）のセットアップ手順プレースホルダーを含める
- バージョン固定ファイル（`.python-version`・`.nvmrc`・`.tool-versions`）への言及を含める
- 主要コマンドのプレースホルダーを含める
- `docs/architecture.md` の「環境設定戦略」セクションに沿った内容にする

**フォーマット**:
```markdown
---
inclusion: always
---

# 技術スタック・環境設定

## 技術スタック
<!-- 例: Python 3.12 / uv / pytest -->
[言語・フレームワーク・主要ライブラリを記入]

## バージョン管理
[バージョン固定ファイルへの言及]

## セットアップ手順
### macOS / ### Windows / ### Linux

## 主要コマンド
[ビルド・テスト・リントコマンドを記入]
```

### 4. structure.md

**責務**:
- リポジトリのディレクトリ構造を Kiro に伝え、ファイル配置の意思決定を支援する
- `docs/repository-structure.md` の詳細版ではなく、Kiro エージェントが参照しやすい要約版を提供する

**実装の要点**:
- プレースホルダーでプロジェクト固有の構造を記入できる形式にする
- テンプレートの基本構造（`.kiro/`・`.steering/`・`docs/`）は予め含めておく
- プロジェクト固有のディレクトリ（`src/`・`tests/` 等）はプレースホルダーで示す

**フォーマット**:
```markdown
---
inclusion: always
---

# リポジトリ構造

## ディレクトリ構造
[ツリー形式で記入]

## 主要ディレクトリの役割
[各ディレクトリの役割を記入]
```

## データフロー

### Kiro エージェントへの注入フロー

```
1. ユーザーが Kiro でプロジェクトを開く
2. Kiro が .kiro/steering/ を検出
3. inclusion: always の4ファイルを全会話コンテキストに注入
4. Kiro エージェントが方針・スタック・構造を把握した状態でユーザーを支援
```

### テンプレート使用フロー

```
1. ユーザーがこのリポジトリを "Use this template" で複製
2. product.md / tech.md / structure.md に自プロジェクト情報を記入
3. process.md はそのまま（ハーネス共通のため編集不要）
4. Kiro でプロジェクトを開き、steering が反映されることを確認
```

## テスト戦略

このプロジェクトはテキストファイル（Markdown）のみで構成されるため、自動テストは対象外。品質確認は以下の手動チェックで行う:

- 各ファイルに `inclusion: always` フロントマターが存在するか
- `<!-- 例: ... -->` 形式の記入ガイドが含まれているか
- プレースホルダーが何を記入すべきか明確か
- ファイル単体で役割が理解できるか

受け入れテストは `docs/architecture.md` の「受け入れテスト」セクションで定義されているシナリオ1（steering コンテキストの確認）を参照する。


## 実装の順序

1. `process.md` を作成（ハーネス共通のルールブック）
2. `product.md` を作成（プロダクト定義プレースホルダー）
3. `tech.md` を作成（技術スタック・環境設定プレースホルダー）
4. `structure.md` を作成（リポジトリ構造プレースホルダー）
5. 各ファイルの品質基準チェック（`docs/development-guidelines.md` 参照）
6. `docs/repository-structure.md` の完成形ツリーとの整合性確認

## 技術的判断と根拠

| 判断 | 根拠 |
|------|------|
| `inclusion: always` を全ファイルに設定 | `docs/architecture.md` および `docs/glossary.md` の定義に従う |
| process.md にプレースホルダーを含めない | ハーネス共通のルールのため、プロジェクト固有の記入は不要 |
| devcontainer への言及を含めない | 設計指針2（devcontainer 非依存）に従う |
| built-in specs の使用禁止を process.md に明記 | 設計指針3に従い、ユーザーが誤って使わないよう明示的に記載する |
