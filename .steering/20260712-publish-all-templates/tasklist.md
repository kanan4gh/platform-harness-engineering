### 学んだこと

**技術的な学び**:
- **CI/CDワークフローの変更は「マージ前に実挙動検証できる」**。`gh workflow run --ref <feature-branch>` でブランチ上のワークフロー定義を実行できる。今回は「テンプレート内容に差分がない」性質を利用し、配布先リポジトリを汚さずに配布経路(導出→matrix→clone→rsync→差分判定)を実際に走らせて検証できた。**副作用のある処理でも、no-opになる条件を選べば安全に実挙動検証できる**という一般則
- GitHub Actionsで`${{ }}`をrun内に直接展開するのはスクリプトインジェクションの経路になる(値がユーザー入力由来の場合)。`env:`に渡してシェル変数として参照するのが規約。今回は`-d`チェックで実害経路はなかったが、規約に沿う形に修正した
- 設定の「ハードコード」は、対象が1つのうちは無害でも、2つ目が増えた瞬間に「片方を忘れる」事故に化ける。**列挙対象が増えうる箇所は動的導出にしておく**(今回は`*-template/`走査)ことで、将来のテンプレート追加時にワークフロー修正が不要になった

**プロセス上の改善点**:
- 「v0.6.0のリリース時にcodexが自動配布されなかった」という**実際の運用上の不便が起点**でIssue化→SDDフローで恒久対処できた。手作業での回避(手動発火)をその場しのぎで終わらせず、原因(ハードコード)を潰す判断が有効だった
