# 設計書

## 実装アプローチ

`kiro-template/ONBOARDING.md` を 1 ファイルで作成する。読者が上から順に操作することで「セットアップ完了 → 各機能の疎通確認 → SDD ワンサイクルで締め」という流れで、ハーネスが自分のプロジェクトで正しく動くことを確認できる構成にする。

**読者の前提**: Kiro の基本操作と SDD の概念は知っている。ハーネスの機能・使い方が初めて。

## 変更対象ファイル

| ファイル | 変更種別 | 内容 |
|---------|---------|------|
| `kiro-template/ONBOARDING.md` | 新規作成 | オンボーディングドキュメント本体 |

## ドキュメント構成

```
# Kiro ハーネス オンボーディングガイド

## 1. このハーネスとは
  - 三層構造の概観（hooks / steering / agents）
  - 各機能の呼び出し方一覧（チートシート）

## 2. セットアップ
  - ステップ1: テンプレートを複製する
  - ステップ2: steering ファイルをカスタマイズする（product / tech / structure）
    - tech.md 記入例: Windows + Python（devcontainer なし）

## 3. 動作確認（ハーネスが正しく動いているか確認する）
  - 確認0: 環境が正しく認識されているか
      → Kiro に「このプロジェクトで Python スクリプトを実行するコマンドを教えてください」と聞く
      → Windows + uv/pip の正しいコマンドが返れば OK
  - 確認1: always steering が読まれているか
      → 「開発プロセスのルールを教えてください」と聞く
  - 確認2: フック（add-feature）を起動する
      → 「add-feature を実行してください」と依頼する
  - 確認3: manual steering（#skill-sdd-guide）を参照する
      → 「#skill-sdd-guide タスクのスキップルールを教えてください」
  - 確認4: カスタムエージェント（@doc-reviewer）を呼び出す
      → 「@doc-reviewer README.md をレビューしてください」
  - 確認5: カスタムエージェント（@implementation-validator）を呼び出す
      → 「@implementation-validator 現在のコードを検証してください」

## 4. SDD でワンサイクル回す（総仕上げ）
  - 前提: GitHub で Issue を作成する
  - 「add-feature を実行してください」でスペック作成フローを起動
  - requirements → design → tasklist を Kiro と一緒に作成
  - 実装（Python / Windows コマンドで動作確認）
  - PR → マージ

## 付録: 注意事項・よくある質問
  - Kiro が Spec mode を提案してきたら断る
  - フックは Hook UI から起動できない（チャット経由）
  - tasklist-check フックについて
```

**ストーリーのポイント**:
- 第2章の `tech.md` 記入でまず環境を Kiro に教える
- 第3章「確認0」でその環境認識が正しいかを最初に確かめる（ここで失敗したら `tech.md` を修正）
- 確認1〜5 でハーネスの各機能を順番に疎通確認
- 第4章で全機能を使って実際に1サイクル回す「総仕上げ」

## 文体・スタイル方針

- **語調**: です・ます調（社内向け丁寧体）
- **コードブロック**: チャットに貼り付ける文言は必ずコードブロックで表示する
- **スクリーンショット**: なし（メンテナンスコストが高いため、テキストで操作を記述）
- **補足ボックス**: `> **注意**:` や `> **ポイント**:` で重要事項を強調
- **長さ**: 通読 15〜20 分を目安（長すぎない）

## Kiro 固有の注意事項（必ず記載するもの）

1. Spec mode への誘導を断る方法（「No」を選ぶ）
2. フックは Hook UI から実行できない → チャットで「X を実行してください」と依頼する
3. `userTriggered` フックは現バージョンでチャット経由のみ起動可能
