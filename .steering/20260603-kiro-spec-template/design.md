# 設計書

## 実装アプローチ

`kiro-template/.steering/_template/` ディレクトリを作成し、3つのテンプレートファイルを配置する。

各テンプレートは以下の設計方針に従う:
- Kiro ユーザーが「何を書けばよいか」が一目で分かる `<!-- 例: ... -->` ガイドを含める
- 過度に複雑にせず、プレースホルダーを埋めるだけで使えるシンプルな構造にする
- `process.md` で定義した SDD フローと整合した順序・構成にする

## ディレクトリ構造

```
kiro-template/
└── .steering/
    └── _template/
        ├── requirements.md   # 新規作成
        ├── design.md         # 新規作成
        └── tasklist.md       # 新規作成
```

## 変更対象ファイル

| ファイル | 変更種別 | 内容 |
|---------|---------|------|
| `kiro-template/.steering/_template/requirements.md` | 新規作成 | 要求仕様テンプレート |
| `kiro-template/.steering/_template/design.md` | 新規作成 | 設計テンプレート |
| `kiro-template/.steering/_template/tasklist.md` | 新規作成 | タスクリストテンプレート |

## 各ファイルの設計

### requirements.md の設計

```
# 要求内容（タイトル固定）

## 概要
[1〜2文のプレースホルダー + 例]

## 関連 Issue（必須）
[Issue URL 記入欄 - 必須と明示]

## ユースケースの軸
[「誰が・何を・できるようになるか」1文プレースホルダー + 例]

## 実装対象の機能
[機能リスト - 複数機能に対応できるよう複数セクション用意]

## 受け入れ条件
[- [ ] 形式チェックリスト]

## スコープ外
[除外事項のリスト]

## 参照ドキュメント
[docs/ への固定リスト]
```

### design.md の設計

Claude Code 用のテンプレートはアーキテクチャ概要・エラーハンドリング・セキュリティ等、コード向けの詳細セクションを含む。Kiro 向けテンプレートはシンプルにし、ユーザーが本質的な設計事項に集中できるようにする。

```
# 設計書（タイトル固定）

## 実装アプローチ
[アプローチを自由記述]

## 変更対象ファイル
[ファイルリスト]

## 技術的判断と根拠
[判断・根拠テーブルまたは自由記述]

## 実装の順序
[番号付き手順リスト]
```

### tasklist.md の設計

Claude Code の `.claude/skills/steering/templates/tasklist.md` と同等の「タスク完全完了の原則」セクションを含める。ただし Kiro 向けなのでコマンド例は技術スタックに依存するプレースホルダーとする。

フェーズ構成:
- フェーズ1: 実装（機能固有タスク）
- フェーズ2: 品質チェック（テスト・リント）
- フェーズ3: ドキュメント更新
- 振り返りセクション（実装後に記入）
- リリース判断セクション

## 技術的判断と根拠

| 判断 | 根拠 |
|------|------|
| design.md をシンプルに保つ | Kiro 向けはコードアーキテクチャではなくファイル変更・設計意図の記述が主。複雑なセクションは混乱を招く |
| tasklist.md に完全完了の原則を含める | `process.md` でタスク完全完了を定義したが、ユーザーが個別テンプレートを見るときに原則を再確認できるようにする |
| `<!-- 例: ... -->` ガイドを全ファイルに含める | `docs/development-guidelines.md` の品質基準に従う |
| 参照ドキュメントは固定リストを提供 | 毎回同じ `docs/` を参照するため、テンプレートに含めておく |
