# 設計書

## 変更方針

ドキュメント修正のみ。コードやフックの変更はなし。

主旨：「手動で steering を書く」フローを「`setup-project` を実行する」フローに置き換え、初回セットアップの入口を一本化する。

```
### 3. steering ファイルをカスタマイズする（必須）
```

### 設計の要点

- `product.md` / `tech.md` / `structure.md` の手動編集手順はすべて削除
- `setup-project` が steering 更新も担うことをひとこと明示
- フック呼び出し方法（チャットで「setup-project を実行してください」）を明記
- Step 4 の見出しを「機能スペックを作成する」→「最初の機能を実装する」に変更し `add-feature` への接続を明示

## ONBOARDING.md の変更設計

### 現状

```
## 2. セットアップ
### ステップ 1: テンプレートを入手してリポジトリを作成する
### ステップ 2: steering ファイルをカスタマイズする
（product.md / tech.md / structure.md の手動編集手順、記入例コードブロック）
```

### 変更後

```
## 2. セットアップ
### ステップ 1: テンプレートを入手してリポジトリを作成する
### ステップ 2: setup-project でプロジェクトを初期化する

チャットで：
    setup-project を実行してください

Kiro が以下を対話形式で行います：
1. プロダクト概要・目的・ターゲットユーザーをヒアリング
2. docs/ 配下に 6 ファイルを順番に作成（各ファイルでユーザー確認あり）
3. .kiro/steering/ の product.md / tech.md / structure.md を実際の内容で更新

> 手動で steering ファイルを編集したい場合は、付録を参照してください。
```

### 設計の要点

- tech.md の記入例コードブロック（Windows + Python サンプル）は主フローから削除し、付録「よくある質問」セクションに移動
- `setup-project` の 3 ステップ（ヒアリング → docs/ 作成 → steering 更新）を箇条書きで可視化
- 手動編集の逃げ道は付録として残す（上級者・既存プロジェクト移行者向け）
- Section 3「動作確認」への導線は変更なし（setup-project 完了後そのまま続く）
