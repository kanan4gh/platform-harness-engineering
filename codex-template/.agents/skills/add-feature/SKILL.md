# 新機能の追加ワークフロー（計画承認後・完全自動実行モード）

**重要:** このワークフローの承認ゲートは**ステップ4.5（計画の承認）の1箇所のみ**です。そこまでは調査・計画を進めて承認を求め、**承認された後は、ユーザーの介入なしに完了まで完全に自動で実行**します。承認後の各ステップは完了後ただちに次のステップへ移行し、途中でユーザーに確認を求めたり作業を中断したりしないでください（「計画は承認必須・実装は自動」の原則）。

## ステップ0: 作業の隔離（フィーチャーブランチ）

main の作業ツリーを汚さないための担保。**このステップ以降の全作業はフィーチャーブランチ上で行う。**

1. 関連する GitHub Issue を確認する。存在しない場合は `gh issue create` で作成し、Issue 番号を控える。
2. 機能名から**英語 kebab-case のタスク名**を決める（例: 「ユーザープロフィール編集」→ `user-profile-edit`）。作業スペックのディレクトリ名（`.steering/[日付]-[タスク名]/`）にも同じタスク名を使う。
3. `git checkout -b feature/[タスク名]` でフィーチャーブランチを作成して切り替える。
4. ブランチ作成に失敗した場合（ブランチ重複等）はユーザーに報告して中断する。自動リネーム等で暗黙に進めない。

**絶対ルール**: **main ブランチ上で実装を始めないこと。**

## ステップ1: 準備

1. 現在のタスクコンテキストを確立する: 機能名 / 日付（YYYYMMDD 形式）/ 作業スペックのパス `.steering/[日付]-[タスク名]/`
2. `.steering/_template/` をコピーして上記ディレクトリを作成する（requirements.md / design.md / tasklist.md の3ファイル）。

## ステップ2: プロジェクト理解

1. `AGENTS.md` の各セクション（プロダクト定義・技術スタック・リポジトリ構造）を前提として確認する。
2. `docs/` 配下の永続ドキュメントを読み、関連する設計思想やアーキテクチャを理解する。

## ステップ3: 既存パターンの調査

1. 「[機能名]」に関連する既存の実装パターン・命名規則・類似コンポーネントをコードベースから検索・分析する。
2. 発見したパターンを設計（ステップ4）の入力として使う。

## ステップ4: 計画フェーズ（作業スペックの記入）

1. ステップ2〜3の結果に基づき、`requirements.md`（関連 Issue URL を必ず記載）→ `design.md` → `tasklist.md` の順に記入する。
2. `tasklist.md` は実装可能なタスクのみをフェーズ分け・実装順序・サブタスク付きで詳細化する（「将来やるかも」「検討中」は含めない）。design.md の1コンポーネントが複数ファイルに及ぶ場合は、**1タスク=1ファイル1変更**の粒度に分解する（分解漏れ防止）。

## ステップ4.5: 計画の承認（このワークフロー唯一の承認ゲート）

1. `design.md` の要点を計画として提示する: 実装アプローチ / 変更対象ファイル / tasklist.md のタスク一覧 / 検証方針。
2. 次の選択肢を提示してユーザーの回答を待つ: 「承認して実装開始 / 修正を指示する / 中止する」。
3. **承認された場合**: 以降は無停止で実行する。ただちにステップ5へ進む。
4. **修正を指示された場合**: 指示を作業スペック（requirements.md / design.md / tasklist.md）に反映し、再度この承認ゲートを実行する。

## ステップ5: 実装ループ（tasklist.md の完全消化）

**このステップは、`tasklist.md` の全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

1. `tasklist.md` を読み込む。
2. 未完了タスク（`- [ ]`）が存在しなければ、ただちに**ステップ6**へ進む。存在すれば次へ。
3. **先頭にある未完了タスク**を1つ特定し、完了させるために必要な実装作業を実行する。
4. 実行したタスクが完了したら、`tasklist.md` の該当行を `[ ]` から `[x]` に**即座に**更新する（完了マークは操作の実行後に付ける）。
5. 1に戻り、処理を繰り返す。

### 実装ループ内の例外処理ルール

- **ルールA: タスクが大きすぎる場合** — タスクをより小さな複数のサブタスクに分割する。元のタスクを削除し、その場所に新しいサブタスク（`[ ]` 付き）を挿入してループを継続する。
- **ルールB: 技術的理由でタスクが不要になった場合** — 実装方針の変更・アーキテクチャの変更・依存関係の変更など、明確な技術的理由がある場合にのみ、`- [x] ~~タスク名~~（理由: [具体的な技術的理由]）` の形式で更新してループを継続する。
- **❌ 絶対禁止**: 未完了タスクを「後でやる」「別タスクにする」などの理由で意図的にスキップすること。理由なく未完了タスクを放置してループを終了させること。ユーザーに判断を仰ぐこと。

## ステップ6: 3段検証

`tasklist.md` の全タスクが完了したことを最終確認してから、以下の3段を順番に実行する。機械的に落ちるものを先に潰し、コストの高い LLM 検証を後段に置く構成である。

**各段の失敗は「修正して同じ段を再実行」。実装コードを変更する修正を行った場合は段1からやり直すこと。**

### 段1: 静的検証

1. `AGENTS.md` の技術スタックセクションに定義されたテスト・リント・型チェックコマンドを順番に実行し、すべてパスすることを確認する。
2. エラーが発生した場合は、問題を分析し、修正してから再度この段を実行する。

### 段2: 実挙動検証

1. 変更種別に応じて観察方法を選ぶ:

| 変更種別 | 観察方法 |
|---------|---------|
| 実行可能コード | 実際に起動・実行し、入出力を観察する |
| 設定・定義ファイル | 定義を実データに対して1回適用し、成果物を観察する |
| 純ドキュメント（docs/ 等のみ） | 観察対象がないため段2をスキップし、その旨を `tasklist.md` に記録する |

2. テスト通過だけで済ませず、必ず実挙動（または成果物）を観察すること。

### 段3: スペック準拠検証

1. `.codex/agents/implementation-validator.toml` のサブエージェントに、作業スペック `.steering/[日付]-[タスク名]/` と実装の整合性検証を委譲する。サブエージェント機構が利用できない環境では、同ファイルの検証観点（要求の反映・設計との整合・タスク消化の妥当性）をメインエージェントが自分で適用する。
2. 指摘に対応する。スペック乖離は実装を修正するか、意図的な乖離として `tasklist.md`（および必要なら `design.md`）に理由を記録する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 振り返りとドキュメント更新

1. `tasklist.md` の「実装後の振り返り」セクションに記入する: 実装完了日 / 計画と実績の差分 / 学んだこと / 次回への改善提案 / リリース判断。
2. 今回の変更がプロジェクトの基本設計やアーキテクチャに影響を与える場合、`docs/` 内の関連する永続ドキュメントを更新する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ8に進むこと。**

## ステップ8: PR 作成

1. 全変更をコミットする（Conventional Commits 形式。本文に `Closes #[Issue番号]` を含める）。**完了マークやコミットは操作を実行した後に記録する**（記録と実態を乖離させない）。
2. ブランチを push し、`gh pr create` で PR を作成する（本文: 概要 / 変更理由 / 変更内容 / テスト結果 / 関連 Issue）。
3. PR の URL をユーザーに報告する。マージはユーザーの判断に委ねる。

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。

- ステップ0: フィーチャーブランチ上で作業している。main に直接コミットしていない。
- ステップ4.5: 計画（design.md / tasklist.md）がユーザーに承認されている。
- ステップ5: `tasklist.md` の全てのタスクが完了状態（`[x]` または正当な理由でスキップ）になっている。
- ステップ6: 3段検証をすべてパスしている（段2は純ドキュメントの場合スキップを記録済み）。
- ステップ7: `tasklist.md` に振り返りが記載されている。
- ステップ8: PR が作成され、URL が報告されている。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
