# 初回プロジェクトセットアップ

`docs/` 配下の6つの永続ドキュメントを、承認ゲート付きで順番に作成します。

## 実行前の確認

1. `docs/ideas/` 配下のマークダウンファイルを全て読む（壁打ち・ブレインストーミングの成果物があれば作成の元にする）。
2. ファイルがなければ、プロダクトの概要・目的・ターゲットユーザーをユーザーに質問する。

## 承認ゲート（全ドキュメント共通）

各ドキュメントの作成後、必ず次の選択肢を提示してユーザーの回答を待つこと:

1. **承認して次へ**
2. **修正を指示する** — 修正点を聞き、反映後に再度この確認を行う
3. **このドキュメントをスキップ** — 既定テンプレートのまま残し、次のステップへ

ただし `product-requirements.md` は以降すべてのドキュメントの土台になるため、**スキップ選択肢は提示しない**（承認/修正の2択）。承認を得る前に次のドキュメントの作成を始めてはいけません。

## 手順

以下の順に1ファイルずつ作成し、各ファイルで承認ゲートを実行する:

1. `docs/product-requirements.md` — プロダクト要求定義書（ユーザーストーリー・受け入れ条件・非機能要件・成功指標）
2. `docs/functional-design.md` — 機能設計書（設計指針・システム構成・コンポーネント設計）
3. `docs/architecture.md` — 技術仕様書（層構造・技術的制約・テスト戦略）
4. `docs/repository-structure.md` — リポジトリ構造定義書（ディレクトリツリー・配置規則）
5. `docs/development-guidelines.md` — 開発ガイドライン（コーディング規約・PRフロー・テスト戦略）
6. `docs/glossary.md` — ユビキタス言語定義（用語集）

各ドキュメントは既存のもの（先に承認された分）と矛盾しないように作成する。

## 仕上げ

6ファイルの作成後、`AGENTS.md` の以下のセクションを実際のプロジェクト内容で更新する:

- プロダクト定義セクション（プロダクト名・目的・docs/ 一覧）
- 技術スタックセクション（技術スタック・OS 別セットアップ手順・主要コマンド）
- リポジトリ構造セクション（プロジェクトのディレクトリ構造）

完了時に、作成したドキュメントの一覧と今後の使い方（会話でのドキュメント編集・add-feature スキルでの機能追加）を報告する。
