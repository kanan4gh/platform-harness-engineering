## 開発プロセス原則

### スペック駆動開発（SDD）フロー

全ての機能開発・バグ修正は以下のフローで行う:

```
1. ドキュメント確認  : docs/ の永続ドキュメントで方針を確認
2. Issue 作成       : GitHub Issue を作成（`gh issue create` コマンド）
3. スペック作成     : .steering/_template/ をコピーして YYYYMMDD-xxx/ を作成
4. 要求定義         : requirements.md を記入（Issue URL を必ず記載）
5. 設計             : design.md を記入
6. タスク計画       : tasklist.md を記入
7. 実装             : tasklist.md のタスクを順番に実行・チェック
8. PR 作成・マージ  : フィーチャーブランチ → PR → main へマージ
9. リリース         : 必要に応じて GitHub Release を作成
```

### 作業スペックの使い方

**ディレクトリ命名規則**:

```
.steering/
├── _template/               # テンプレート（直接編集しない）
├── 20250115-add-login/      # 機能実装単位（_template/ をコピーして作成）
│   ├── requirements.md      # 要求仕様（関連 Issue URL を必須記入）
│   ├── design.md            # 実装設計
│   └── tasklist.md          # タスクリスト（進捗を随時チェック）
└── 20250120-fix-auth-bug/
    └── ...
```

命名規則: `YYYYMMDD-[タスク名（英小文字・ハイフン区切り）]`

**スペック作成手順**:

1. GitHub Issue を作成する（`gh issue create`）
2. `.steering/_template/` をコピーして `YYYYMMDD-[タスク名]/` として配置する
3. `requirements.md` を記入し、**ユーザーの確認を得てから** `design.md` に進む
4. `design.md` を記入し、**ユーザーの確認を得てから** `tasklist.md` に進む
5. `tasklist.md` を記入し、**ユーザーの確認を得てから**実装を開始する

> **重要**: 各ドキュメントは1ファイルずつ作成する。ユーザーが確認・承認するまで次のステップに進まない。
> **例外**: `add-feature` スキル（ワークフローセクション参照）で作業する場合、上記3〜5の個別確認は**ステップ4.5「計画の承認」の1箇所に統合**される（3ファイルを作成してから計画として一括提示する）。

### tasklist.md の管理ルール

- タスク完了時（操作の実行後）に `[ ]` → `[x]` へ**即座に**更新する
- **完了マークは操作の実行後に付ける**（実行予定の段階で付けない。コミット・PR 作成など、マークの後に実行しがちな操作は特に注意。マークと実態の乖離はフックでは検出できない）
- **全タスクが `[x]` になるまで実装を継続する**
- 「時間の都合により後回し」は禁止
- 技術的理由でスキップする場合のみ `- [x] ~~タスク名~~（理由: 具体的な技術的理由）` の形式でスキップできる
- タスクが大きすぎる場合はサブタスクに分割して1つずつ完了させる

**フックによる担保**: このテンプレートには Stop フック（`.codex/hooks.json` + `.codex/hooks/check_tasklist_complete.py`）が含まれており、最新の作業スペックの tasklist.md に `- [ ]` が残っているとセッション終了がブロックされ、継続が促される（プロジェクトを trust している場合のみ有効）。ブロックされた場合は、タスクを完了させるか、上記の分割・技術的スキップを適用すること。

### ドキュメント管理

**永続ドキュメント（`docs/`）**:
- プロジェクト全体の「何を作るか」「どう作るか」を定義する
- 頻繁に更新しない（変更は方針転換のシグナル）
- 実装後に設計と実態が乖離していれば更新する

**作業スペック（`.steering/`）**:
- 特定の作業に特化したドキュメント
- 作業ごとに新規作成し、履歴として保持する（`.gitignore` に含めない）

### PR・リリースフロー

**PR 作成**:
- `main` へ直接コミットしない（機能実装コードも含む）
- フィーチャーブランチ → PR → マージ の手順を踏む

**スタック PR の扱い**（スタック PR = ベースが別のフィーチャーブランチの PR。マージ順事故の温床のため原則避ける）:
- **可能な限り、ベースのマージを待って通常 PR（base: main）にする**。複数の変更を並行させる場合は、PR 間で変更ファイルを重複させず任意順でマージ可能にする
- やむを得ずスタック PR を使う場合は、**必ずベース側から順にマージする**（PR 本文の依存注記だけでは機構的に防げない）
- **マージ後は「デフォルトブランチに期待する変化が実在するか」を検証する**（例: 変更したはずの設定値を main で確認、Closes が発火して Issue が閉じたか確認）

**リリース**:
- PR マージ後、必要に応じて `gh release create` で GitHub Release を作成する
- リリース前に関連 Issue をクローズする（`gh issue close <番号>`）

### LLM 向け定義ファイルの記述規約

フック・スキル（SKILL.md）・エージェント定義は LLM が実行する「手順書」であり、曖昧さが実行者依存の挙動差を生む:

- **設計の具体形を転記する**: 設計書にファイルパターン・コマンド・正規表現等の具体形がある場合、定義ファイルにはその具体形をそのまま転記する（散文で言い換えない。散文だけだと実行者が広いパターンを選び、規約外のファイルを誤処理する事故が起きる）
- **既存ファイルの用語に引っ張られない**: 既存ファイルへ追記する際、既存用語が設計書と食い違っている場合は設計書の用語を優先する（用語の正典はドキュメントにある）

### 記憶層の運用

Codex CLI のネイティブ Memories 機構を記憶層として利用する（自作の `memory/` ディレクトリは作らない）:

- **記憶させるべきもの**: セッションをまたいで必要になるユーザー決定・まだドキュメント化されていない作業方針
- **記憶させないもの**: リポジトリが既に記録しているもの（コード・docs/・git 履歴・.steering/）
- Memories が利用できない環境（EEA・UK・スイス等の地域制限、または機能無効時）では、セッション横断で必要な知見を `AGENTS.md` のプロダクト定義セクションに記録して代替する

## ワークフロー

<!-- このセクションはハーネス共通のワークフロー定義です。編集しないでください。 -->

定型ワークフローの詳細手順は `.agents/skills/` 配下のスキルに定義されている。以下の依頼を受けたら、該当スキルを読み込んでその手順に従うこと。

### setup-project

「setup-project を実行してください」「プロジェクトをセットアップして」と依頼された場合、`.agents/skills/setup-project/SKILL.md` の手順に従う。

要点: `docs/` 配下の 6 ドキュメントを1ファイルずつ作成し、各ファイルで承認ゲート（承認して次へ / 修正を指示する / スキップ。PRD はスキップ不可）を実行する。最後に `AGENTS.md` のプロダクト定義・技術スタック・リポジトリ構造セクションを更新する。

### add-feature

「add-feature を実行してください」「新機能を追加して」と依頼された場合、`.agents/skills/add-feature/SKILL.md` の手順に従う。

要点: フィーチャーブランチ必須（main で実装しない）。**承認ゲートは「計画の承認」の1箇所のみ**で、承認後はユーザーの介入なしに完了まで自動実行する（実装ループ → 3段検証 → 振り返り → PR 作成）。

### distill

「distill を実行してください」「蒸留して」と依頼された場合、`.agents/skills/distill/SKILL.md` の手順に従う。

要点: `.steering/` の振り返りから知見を抽出し、環流3基準（普遍性・再現性・抽象性）で分類して platform-harness への環流候補を提示する。
