# プロジェクト用語集 (Glossary)

**更新日**: 2026-06-03

このドキュメントは、プロジェクト内で使用される用語を統一的に定義する。ドキュメント間の認識齟齬を防ぐことが目的である。

---

## ハーネスエンジニアリング用語

### ハーネス

**定義**: 「エンジンを動かすために必要な、エンジン以外のすべて」。AI 駆動開発においては LLM（エンジン）を有効に動かすすべての仕組みを指す。

**説明**: テストハーネス（テスト対象を動かすための環境・入力・検証の仕組み）から概念を拡張した定義。文脈注入・プロセスの型・記憶・検証・フィードバックで構成される。

**関連用語**: プラットフォームハーネス、プロジェクトハーネス、ハーネスエンジニアリング

**英語表記**: Harness

---

### ハーネスエンジニアリング

**定義**: LLM を有効に動かすハーネスを設計・構築・改善する活動。「エンジニアがコードを書く」から「エンジニアがハーネスを設計し、LLM がコードを書く」への役割転換を表す概念。

**関連用語**: ハーネス、スペック駆動開発

**英語表記**: Harness Engineering

---

### プラットフォームハーネス

**定義**: プロジェクトをまたいで繰り返し使われ、経験を通じて継続的に進化するハーネスの層。

**説明**: このリポジトリ（`platform-harness-for-kiro`）がプラットフォームハーネスに相当する。steering ファイルのテンプレート・スペックテンプレート・開発プロセス原則等が含まれる。

**対義語**: プロジェクトハーネス

**英語表記**: Platform Harness

---

### プロジェクトハーネス

**定義**: 特定のプロジェクト・作業単位に固有のハーネス。プラットフォームハーネスの「型」に具体的な内容を埋めたもの。

**説明**: `.steering/YYYYMMDD-xxx/` 配下の requirements.md・design.md・tasklist.md がプロジェクトハーネスに相当する。

**対義語**: プラットフォームハーネス

**英語表記**: Project Harness

---

### スペック駆動開発

**定義**: 実装前に要求仕様（spec）を文書化し、それを「唯一の正解」として LLM が実装を行う開発スタイル。コードはスペックのビルド成果物として位置づけられる。

**説明**: 「vibe coding（雰囲気でプロンプトを打つ）」に対する概念。ドキュメント → スペック作成 → 実装 → 検証 → 環流というフローで構成される。

**英語表記**: Spec-Driven Development / SDD

---

### 環流

**定義**: プロジェクトでの経験・失敗・洗練から得た知恵を抽象化し、プラットフォームハーネスコアに書き戻すプロセス。

**説明**: 環流の判断基準は「普遍性・再現性・抽象性」の3点。プロジェクト固有の値（URL・IDなど）を含むものは環流しない。

**関連用語**: 蒸留、プラットフォームハーネス

---

### 蒸留

**定義**: プロジェクトの経験から「次に活かすべき普遍的な知恵」を取り出す作業。環流の前段階。

**関連用語**: 環流

---

### ブートストラップ（フェーズ）

**定義**: あるツール（Claude CLI）を使って、そのツールの後継・代替となるツール（Kiro ハーネス）を設計・実装する初期フェーズ。

**説明**: コンパイラのブートストラップ（旧言語で新コンパイラの初版を作り、その後自分自身でコンパイルするようにする）と同じ構造。本プロジェクトにおいては Claude CLI で Kiro ハーネスを設計・実装する期間がこれに相当する。

**関連用語**: 自走フェーズ、Bootstrap フェーズ

---

### 自走フェーズ

**定義**: Kiro ハーネスが自分自身の改善に使われるようになった状態。Claude CLI 環境がなくても Kiro ハーネスの改善が完結できる状態を指す。

**説明**: 自走フェーズに入ることが「Claude CLI 環境を削除してよい」タイミングの条件となる。ただし削除は必須ではない。

**関連用語**: ブートストラップ（フェーズ）、Bootstrap フェーズ

---

## Kiro 関連用語

### Kiro

**定義**: Amazon が提供するエージェント型 IDE。Spec-Driven Development を中心に設計されており、steering・hooks・specs 等のネイティブ機能を持つ。

**本プロジェクトでの用途**: このハーネスの移植先となる IDE。

**公式サイト**: https://kiro.dev

---

### steering ファイル

**定義**: Kiro エージェントのコンテキストに常時（または条件付きで）注入されるマークダウンファイル。`.kiro/steering/` に配置する。

**説明**: Claude CLI の `CLAUDE.md` に相当する。`inclusionMode` によって注入タイミングを制御できる（always / auto / manual）。

**本プロジェクトでの用途**: `process.md`・`product.md`・`tech.md`・`structure.md` の4ファイルを全て `always` で設定する。

**関連用語**: inclusionMode

---

### inclusionMode

**定義**: Kiro の steering ファイルがコンテキストに注入されるタイミングを制御するフロントマター設定値。

**値の種類**:

| 値 | 説明 |
|----|------|
| `always` | 常時コンテキストに含める |
| `auto` | 関連ファイルが開かれた時に自動で含める |
| `manual` | 明示的に指定された時のみ含める |

**本プロジェクトでの使用**: 全 steering ファイルを `always` に設定する。

---

### hooks（Kiro）

**定義**: Kiro のトリガー機能。ファイル保存・作成・削除等のイベントに応じてエージェントへの指示を自動実行する。`.kiro/hooks/` に JSON ファイルで定義する。

**説明**: Claude CLI の `settings.json` hooks に相当するが API 仕様が異なる。kiro-cli を必要とするコマンドは含めない（設計指針1）。

**関連用語**: kiro-cli

---

### Kiro built-in specs

**定義**: Kiro が提供する内蔵の specs 機能。`.kiro/specs/` ディレクトリに requirements.md・design.md・tasks.md を配置して使う UI 統合機能。

**本プロジェクトでの扱い**: **使用しない**（設計指針3）。AWS AI-DLC（`awslabs/aidlc-workflows`）も明示的に回避しており、動作の不安定さ・仕様変更リスクを理由に除外した。代わりに `.steering/YYYYMMDD-xxx/` 構造を使う。

**関連用語**: .steering/、設計指針3

---

### kiro-cli

**定義**: Kiro が提供するコマンドラインインターフェース。

**本プロジェクトでの扱い**: **使用しない**（設計指針1）。CLI を使いこなせないユーザーの利用を想定するため、全操作を Kiro IDE の GUI 内で完結させる。

---

## プロジェクト構成要素

### 文脈層

**定義**: ハーネスの4層構造の最上位層。Kiro エージェントに「何者か・何をすべきか」を常時伝える層。

**実体**: `.kiro/steering/` 配下の4つのファイル（`process.md`・`product.md`・`tech.md`・`structure.md`）

**関連用語**: プロジェクトハーネス層、自動化層、永続ドキュメント層

---

### プロジェクトハーネス層

**定義**: ハーネスの4層構造の第2層。作業単位で「何を・どう・どの順で」実装するかを定義する層。

**実体**: `.steering/YYYYMMDD-xxx/` 配下の requirements.md・design.md・tasklist.md

**関連用語**: 文脈層、自動化層、永続ドキュメント層

---

### 自動化層

**定義**: ハーネスの4層構造の第3層。定型作業をトリガーで自動実行する層。

**実体**: `.kiro/hooks/` 配下の JSON ファイル群

**関連用語**: 文脈層、プロジェクトハーネス層、永続ドキュメント層

---

### 永続ドキュメント層

**定義**: ハーネスの4層構造の第4層。プロジェクト全体の「北極星」となるドキュメントを保持する層。

**実体**: `docs/` 配下の6ドキュメント（PRD・機能設計書・アーキテクチャ・リポジトリ構造・開発ガイドライン・用語集）

**関連用語**: 文脈層、プロジェクトハーネス層、自動化層

---

### スペック（作業単位スペック）

**定義**: 1つの機能追加・バグ修正等の作業単位に対して作成するドキュメント群（requirements.md・design.md・tasklist.md）。`.steering/YYYYMMDD-xxx/` に配置する。

**注意**: Kiro の「built-in specs」とは別物であり、このプロジェクトでは built-in specs を使わない。

**関連用語**: Kiro built-in specs、プロジェクトハーネス層

---

### 受け入れテスト

**定義**: Kiro ハーネスが実際に Kiro 上で動作するかを確認する手動テスト。初回リリースの唯一のゲート。

**説明**: 自動化できない。Kiro を実際に開き、steering ファイルの反映・スペックフロー・制約（kiro-cli 不使用・devcontainer 不要）・テンプレートのカスタマイズ可否を人間が確認する。詳細は `docs/architecture.md`「テスト戦略」を参照。

---

## 略語・頭字語

### AI-DLC

**正式名称**: AI-Driven Life Cycle（または AI-Driven Learning Content、文脈による）

**本プロジェクトでの使用**: `awslabs/aidlc-workflows` として GitHub で公開されている AWS の AI 開発ライフサイクル向けワークフロー定義。Kiro の built-in specs を使わないことを明示しており、本プロジェクトの設計指針3（specs 不使用）の根拠として参照した。

---

### MCP

**正式名称**: Model Context Protocol

**意味**: LLM が外部システム（GitHub・AWS 等）と連携するためのオープンプロトコル。MCP サーバーを追加することで LLM の「手の届く範囲」を拡張できる。

**本プロジェクトでの使用**: `.kiro/settings/mcp.json` でプロジェクト固有の MCP サーバーを設定する。プラットフォーム共通 MCP はユーザーのグローバル設定で管理する。

---

### PRD

**正式名称**: Product Requirements Document

**意味**: プロダクト要求定義書。「何を作るか」「誰のためか」「成功指標は何か」を定義するドキュメント。

**本プロジェクトでの使用**: `docs/product-requirements.md`

---

### SDD

**正式名称**: Spec-Driven Development

**意味**: スペック駆動開発。

**関連用語**: スペック駆動開発

---

### IDE

**正式名称**: Integrated Development Environment

**意味**: 統合開発環境。本プロジェクトの文脈では Kiro を指すことが多い。
