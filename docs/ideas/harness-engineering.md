# ハーネスエンジニアリング

_2026-05-27 ディスカッションのまとめ_

## ハーネスとは何か

> **「エンジンを動かすために必要な、エンジン以外のすべて」**

従来のソフトウェア開発における「テストハーネス」から概念を拡張した定義。

| | エンジン | ハーネス |
|--|---------|---------|
| **従来（テスト）** | テスト対象のコード | テスト環境・入力・検証の仕組み |
| **AI駆動開発** | LLM（Claude） | LLMを有効に動かすすべての仕組み |

## AI駆動開発におけるハーネスの構成要素

| 要素 | 具体例 | 役割 |
|------|--------|------|
| **文脈の注入** | CLAUDE.md、requirements.md | LLMが何をすべきかを定義する |
| **プロセスの型** | steering スキル、tasklist.md | 作業手順を強制する |
| **記憶** | memory/ | セッションをまたぐ一貫性を保つ |
| **検証** | テスト、ruff、動作確認 | 出力の品質を保証する |
| **フィードバック** | ユーザーの指摘・修正 | ハーネス自体を改善する |

## エンジニアの役割の変化

従来: **エンジニアがコードを書く** → ハーネス（テスト）で検証  
AI駆動: **エンジニアがハーネスを設計する** → LLMがコードを書く

スペック駆動開発において、「スペックを書く行為」そのものがハーネスを設計する行為である。

## ハーネスの2層構造

```
┌──────────────────────────────────────────┐
│        プラットフォームハーネス           │
│  (Platform Harness)                      │
│  プロジェクト横断・再利用・継続的に進化   │
├──────────────────────────────────────────┤
│        プロジェクトハーネス              │
│  (Project Harness)                       │
│  プロジェクト固有・スペックそのもの      │
└──────────────────────────────────────────┘
```

### プラットフォームハーネス（再利用層）

プロジェクトをまたいで繰り返し使われ、経験を通じて継続的に進化するもの。

- **CLAUDE.md 汎用層** — スペック駆動開発の原則
- **スキル** — `steering`、`add-feature`、`review-docs` など
- **テンプレート** — `skills/steering/templates/requirements.md` など
- **memory/** — ユーザー・プロジェクト横断の知識

### プロジェクトハーネス（固有層）

特定のプロジェクト・作業単位に固有のもの。プラットフォームハーネスの「型」に具体的な内容を埋めたもの。

- **requirements.md** — テンプレート（型）+ プロジェクト固有の要求（内容）
- **design.md** — テンプレート（型）+ プロジェクト固有の設計（内容）
- **tasklist.md** — テンプレート（型）+ プロジェクト固有のタスク（内容）

## 進化のサイクル

```
プロジェクトハーネス（個別の経験・失敗・洗練）
        ↓  抽象化・蒸留
プラットフォームハーネス（再利用可能な知恵として結晶化）
        ↓  適用
次のプロジェクトハーネス
        ↓  また洗練...
```

プラットフォームハーネスは **「過去の失敗と洗練の結晶」** として進化していく。

### 進化の具体例

- Hands-on #2b で「ユースケースの軸ずれ」（チャット→MNIST認識）を経験
- → requirements.md テンプレートに「ユースケースの軸を最初に明示する」項目を追加
- → 次のプロジェクトでは同じ軸ずれが起きにくくなる

## ハーネスエンジニアリングのサイクル

```
プラットフォームハーネスコア
（platform-harness リポジトリ）
        │
        │ ① 展開
        ▼
プラットフォームハーネス実装
（CLAUDE.md 汎用層、skills/、memory/）
        │
        │ ② 固有化
        ▼
プロジェクトハーネス
（requirements.md、design.md、tasklist.md）
        │
        │ ③ 実装
        ▼
  具体的な成果物
（コード、ドキュメント、動作確認）
        │
        │ ④ 蒸留
        ▼
プラットフォームハーネスの洗練
（スキル更新、CLAUDE.md 改善、ideas/）
        │
        │ ⑤ 環流
        ▼
プラットフォームハーネスコア ← 進化した状態で戻る
```

| フェーズ | 内容 |
|---------|------|
| **① 展開** | プラットフォームハーネスコアを新しいプロジェクトに持ち込む |
| **② 固有化** | 技術スタック・目的・制約を加えてプロジェクト専用に仕立てる |
| **③ 実装** | プロジェクトハーネス（スペック）を作りながら LLM と協働して成果物を生む |
| **④ 蒸留** | 実装の経験から「次に活かすべき知恵」を取り出す |
| **⑤ 環流** | 洗練された知恵をコアに書き戻し、次のすべてのプロジェクトが恩恵を受ける |

## `platform-harness` という名前について

プラットフォームハーネスコア（`platform-harness` リポジトリ）は、当初 `claude-guidelines` という名前で作られた。しかし、ハーネスエンジニアリングの概念が言語化された 2026-05-27 のディスカッションを経て、`platform-harness` に改名した。

**`claude-guidelines` の問題：**
- 主語がツール（Claude）になっており、LLM に従属した印象を与える
- LLM が変わると名前が腐る
- 何が入っているかが名前から分からない

**`platform-harness` が適切な理由：**
- 何であるか（プラットフォーム層のハーネス）が名前から分かる
- LLM に依存しない概念であることが伝わる
- 今日言語化した概念と一致する

この改名自体が「プロジェクトの経験を通じてプラットフォームハーネスが進化した」一例である。

## ハーネスの構成要素（詳細）

プラットフォームハーネスは以下の層で構成される：

```
プラットフォームハーネス
├── 文脈層:  CLAUDE.md、requirements.md テンプレート等
├── プロセス層: skills/（steering、add-feature 等）
├── 記憶層:  memory/
├── 検証層:  テスト、リンター等
└── ツール層: MCP サーバー
```

### ツール層: MCP サーバー

MCP（Model Context Protocol）サーバーは LLM の「手の届く範囲」を決める層。MCP サーバーなしでは LLM はテキスト・ファイルのみを扱えるが、MCP サーバーを追加することで外部システム（GitHub、AWS、カレンダー等）と連携できるようになる。

MCP サーバーもプラットフォームハーネスと同じ2層構造で考える：

| | プラットフォーム層 MCP | プロジェクト固有 MCP |
|--|----------------------|-------------------|
| **例** | GitHub、ファイルシステム | AWS MCP（AWS プロジェクト）、Jira（特定チーム） |
| **設定場所** | ユーザー設定（全プロジェクト共通） | `.mcp.json`（リポジトリ固有） |
| **更新契機** | ほぼ変わらない | プロジェクトの技術スタックに応じて |

## 環流ルール

### 判断基準: プラットフォームハーネスコアに環流すべきか

以下の問いにすべて **Yes** なら環流対象：

1. **普遍性**: どのプロジェクトでも有効か？（特定の技術スタック・クラウドに依存しないか）
2. **再現性**: 同じ失敗・同じ改善が他のプロジェクトでも起こりうるか？
3. **抽象性**: プロジェクト固有の具体的な値（URL・モデルID・リージョン等）を含まないか？

### 環流プロセス

```
1. フィーチャーブランチのマージ後、振り返りで洗練結果を tasklist.md に記録
2. プロジェクトの節目に棚卸しを実施（上記判断基準で分類）
3. platform-harness リポジトリを clone して変更を加える
4. PR を作成・マージ
5. このプロジェクトの CLAUDE.md の UPDATED 日付を更新
```

## このリポジトリにおける対応

| | プラットフォームハーネス | プロジェクトハーネス |
|--|------------------------|---------------------|
| **具体例** | CLAUDE.md 汎用層、skills/、memory/ | .steering/ 配下の各ファイル |
| **テンプレート** | skills/steering/templates/ ← **型** | .steering/YYYYMMDD-xxx/ ← **型+内容** |
| **寿命** | プロジェクトをまたいで持続 | 1作業単位で完結（履歴として保持） |
| **更新契機** | 洗練・学びのたびに随時 | 実装のたびに新規作成 |
