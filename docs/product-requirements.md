# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**platform-harness-for-kiro** - Kiro向けスペック駆動開発ハーネス テンプレート

### プロダクトコンセプト
- **概念移植**: Claude CLI ハーネスの構成要素（文脈層・プロセス層・記憶層・ツール層）を、Kiro のネイティブ機能に対応付けて再実装する
- **ワークフロー継続性**: Claude CLI で確立したスペック駆動開発フロー（ドキュメント作成 → 作業計画 → 実装 → 検証 → 環流）を Kiro 上でも維持する
- **テンプレートとしての再利用性**: 新規プロジェクト開始時にこのリポジトリを "Use this template" で複製し、即座にスペック駆動開発を開始できる

### プロダクトビジョン
Kiro を使うエンジニアが、Claude CLI ハーネスで培ったスペック駆動開発のノウハウをそのまま活用できるようにする。
LLM（エンジン）を有効に動かすすべての仕組み（ハーネス）を、Kiro のネイティブ機能（steering・hooks 等）を使って再構築し、ツールが変わっても「エンジニアがハーネスを設計し、LLM が実装する」という開発スタイルを維持する。
CLI 操作や Docker 環境を前提とせず、Kiro IDE の GUI 内で完結するワークフローを実現することで、幅広い技術レベルのエンジニアが利用できるようにする。

### 目的
- Claude CLI ハーネスの各構成要素と Kiro 機能の対応関係を明確にする
- Kiro 上でスペック駆動開発を開始するためのテンプレートリポジトリを提供する
- ハーネスの進化サイクル（プロジェクト経験 → 蒸留 → プラットフォームハーネス改善）を Kiro 環境でも機能させる

## ターゲットユーザー

### プライマリーペルソナ: Claude CLI ハーネス経験者
- Claude Code（Claude CLI）でスペック駆動開発を実践してきたエンジニア
- Kiro へ移行、または Kiro を並行利用したいと考えている
- 現在の課題: Kiro のネイティブ機能（steering/hooks 等）と Claude CLI ハーネスの概念が対応しているのか分からず、移行コストが不透明
- 期待する解決策: Claude CLI ハーネスで使っていたワークフローをそのまま Kiro で再現できるテンプレートと概念マッピング

### セカンダリーペルソナ: Kiro 新規ユーザー
- Kiro を使い始めたエンジニアで、スペック駆動開発のハーネスを整備したい
- Claude CLI の経験はないが、構造化された AI 協働開発フローを求めている

## 成功指標(KPI)

### プライマリーKPI
- **テンプレート完成度**: Claude CLI ハーネスの全構成要素（文脈層・プロセス層・記憶層・ツール層）が Kiro 上で機能すること
- **セットアップ時間**: テンプレートを複製してから最初のスペック駆動開発を開始するまで 10 分以内
- **ワークフロー網羅率**: Claude CLI の主要ワークフロー（setup-project・add-feature・steering・review-docs）に相当する Kiro フローが全て定義されていること

### セカンダリーKPI
- **概念マッピングの明確さ**: Claude CLI ハーネスの各要素と Kiro 機能の対応表が docs/ に記述されていること
- **環流可能性**: Kiro でのプロジェクト経験をプラットフォームハーネスコアに蒸留・環流できる仕組みが整っていること

## 機能要件

### コア機能(MVP)

#### 概念マッピングドキュメント

**ユーザーストーリー**:
Claude CLI ハーネス経験者として、Kiro への移行コストを下げるために、両者の概念対応表が欲しい

**受け入れ条件**:
- [ ] Claude CLI の構成要素（CLAUDE.md・skills/・memory/・.steering/・.mcp.json）と Kiro 機能（steering・specs・hooks・agents・settings）の対応表が `docs/` に存在する
- [ ] 各対応において「完全移植」「部分移植」「代替手段あり」「未対応」が明記されている
- [ ] 移行手順（Claude CLI → Kiro）が step-by-step で記述されている

**優先度**: P0(必須)

---

#### Kiro Steering ドキュメント群（文脈層）

**ユーザーストーリー**:
Kiro ユーザーとして、プロジェクト全体の文脈を LLM に常に提供するために、Kiro の steering 機能を活用した文脈注入の仕組みが欲しい

**受け入れ条件**:
- [ ] `.kiro/steering/` 配下に product.md・tech.md・structure.md のテンプレートが存在する
- [ ] Claude CLI の CLAUDE.md 3 層構造（汎用層・プロダクト固有層・技術スタック固有層）が Kiro steering 形式に変換されている
- [ ] 各 steering ファイルに `inclusionMode`（always/auto/manual）が設定されている

**優先度**: P0(必須)

---

#### Kiro Specs テンプレート（プロジェクトハーネス層）

**ユーザーストーリー**:
Kiro ユーザーとして、機能実装の都度スペック駆動開発フローを実行するために、Kiro の specs 機能を使ったテンプレートが欲しい

**受け入れ条件**:
- [ ] `.kiro/specs/` 配下に requirements.md・design.md・tasks.md のテンプレートが存在する
- [ ] テンプレートが Claude CLI の `.steering/` テンプレートと同等の品質基準を持つ
- [ ] Kiro の specs ワークフロー（requirements → design → tasks → implementation）が文書化されている

**優先度**: P0(必須)

---

#### Kiro Hooks（プロセス自動化層）

**ユーザーストーリー**:
Kiro ユーザーとして、定型的な開発プロセスを自動化するために、Claude CLI の hooks 相当の自動化が Kiro 上で動作して欲しい

**受け入れ条件**:
- [ ] `.kiro/hooks/` 配下に Claude CLI の主要 hooks に相当する設定が存在する
- [ ] 少なくとも「ファイル保存時の型チェック」「テスト実行」に相当する hook が定義されている
- [ ] hooks の設定方法と動作確認手順が文書化されている

**優先度**: P1(重要)

---

#### 永続ドキュメント体系（docs/）

**ユーザーストーリー**:
Kiro ユーザーとして、プロジェクト全体の「何を作るか」「どう作るか」を管理するために、Claude CLI と同等の docs/ 体系が欲しい

**受け入れ条件**:
- [ ] `docs/` 配下に 6 つの永続ドキュメントテンプレートが存在する（product-requirements・functional-design・architecture・repository-structure・development-guidelines・glossary）
- [ ] `docs/ideas/` が壁打ち・ブレインストーミング用として定義されている
- [ ] 各ドキュメントの用途と更新契機が CLAUDE.md 相当のファイルに記述されている

**優先度**: P0(必須)

---

#### セットアップガイド

**ユーザーストーリー**:
新規ユーザーとして、このテンプレートを使って 10 分以内にスペック駆動開発を開始するために、明確なセットアップ手順が欲しい

**受け入れ条件**:
- [ ] README.md にテンプレートの使い方（"Use this template" → カスタマイズ手順）が記述されている
- [ ] Claude CLI ハーネスからの移行手順と、Kiro 新規ユーザー向けの手順が両方記述されている
- [ ] セットアップ完了後に「最初の機能実装スペックを作成する」手順まで案内されている

**優先度**: P0(必須)

---

### 将来的な機能(Post-MVP)

#### Kiro エージェント定義（プロセス層の高度化）

Claude CLI の `skills/`（steering・add-feature・review-docs 等）に相当する Kiro カスタムエージェントを定義する。

**優先度**: P2(できれば)

---

#### 環流プロセスの Kiro 対応

Kiro でのプロジェクト経験をプラットフォームハーネスコアに蒸留・環流するプロセスを、Kiro のワークフローに合わせて再定義する。

**優先度**: P2(できれば)

## 非機能要件

### ユーザビリティ
- テンプレート複製から最初のスペック作成開始まで 10 分以内（Kiro 経験者の場合）
- Claude CLI ハーネス経験者が README を読むだけで概念マッピングを理解できる
- 各テンプレートファイルにコメント（`<!-- 例: ... -->`）で記入ガイドが含まれる

### 保守性
- Kiro の機能更新に伴うテンプレート変更が局所的に行えるよう、ファイルが責務ごとに分離されている
- Claude CLI ハーネスとの対応関係が docs/ に記述されており、両者の同期状態が追跡可能

### 信頼性
- テンプレートファイルはそのままコピーして使えること（未記入のプレースホルダーが明確に区別できる）
- Kiro の steering ファイルが構文エラーを含まないこと

## スコープ外

- Kiro 本体の機能開発・改修（Kiro は Amazon が開発する外部ツール）
- Claude CLI ハーネスの機能追加・改修（別リポジトリで管理）
- 特定プロジェクト向けのカスタマイズ（このリポジトリはテンプレートのみ）
- memory/ の Kiro 完全対応（Kiro に永続メモリ機能がない場合は代替手段を文書化するにとどめる）
- CI/CD パイプラインの整備（プロジェクト固有のため対象外）
