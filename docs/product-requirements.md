# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**platform-harness-engineering** - AI ツール向けスペック駆動開発ハーネス プラットフォーム

### プロダクトコンセプト
- **マルチツール対応**: Kiro・Codex CLI など複数の AI ツール向けハーネステンプレートを一元管理・配布する
- **ハーネス設計の標準化**: SDD（スペック駆動開発）のフロー（ドキュメント作成 → 作業計画 → 実装 → 検証 → 環流）を、どの AI ツール上でも再現できるテンプレートとして提供する
- **テンプレートとしての再利用性**: 新規プロジェクト開始時に各ツール向けテンプレートリポジトリを "Use this template" で複製し、即座にスペック駆動開発を開始できる
- **拡張性**: 新しい AI ツールが登場したとき、`{tool}-template/` を追加するだけで対応範囲を拡大できる

### プロダクトビジョン
使用する AI ツールが何であれ、エンジニアが「ハーネスを設計し、LLM が実装する」スペック駆動開発スタイルを即座に始められるようにする。
Kiro・Codex CLI・Claude Code など各ツールのネイティブ機能（steering・AGENTS.md・CLAUDE.md 等）を活かしたハーネステンプレートを設計・管理・配布するプラットフォームとして、ツールの多様化・進化に対応し続ける。

### 目的
- 各 AI ツールのネイティブ機能と SDD ハーネスの構成要素（文脈層・プロセス層・記憶層・ツール層）の対応関係を明確にする
- 各 AI ツール向けに SDD を開始するためのテンプレートリポジトリを提供する
- ハーネスの進化サイクル（プロジェクト経験 → 蒸留 → プラットフォームハーネス改善）を維持する
- 新ツールへの対応を標準プロセスで追加できる拡張性を持つ

## ターゲットユーザー

### プライマリーペルソナ: SDD ハーネスを整備・運用したいエンジニア
- スペック駆動開発を実践しており、使用する AI ツール向けのハーネスを整備したい
- Kiro・Codex CLI・Claude Code など複数ツールを状況に応じて使い分けている、または移行を検討している
- 現在の課題: ツールごとにハーネスをゼロから設計するコストが高く、SDD フローの一貫性が保ちにくい
- 期待する解決策: ツールのネイティブ機能に対応した、すぐ使えるハーネステンプレート

### セカンダリーペルソナ: 特定 AI ツールの新規ユーザー
- Kiro または Codex CLI を使い始めたエンジニアで、構造化された AI 協働開発フローを求めている
- SDD の経験はないが、ベストプラクティスに沿った開発スタイルを最初から取り入れたい

## 成功指標(KPI)

### プライマリーKPI
- **テンプレート完成度**: 各テンプレートで SDD の全構成要素（文脈層・プロセス層・記憶層・ツール層）が機能すること
- **セットアップ時間**: テンプレートを複製してから最初のスペック駆動開発を開始するまで 10 分以内
- **ツール対応数**: 現在 2 ツール対応（Kiro / Codex CLI）

### セカンダリーKPI
- **ワークフロー網羅率**: setup-project・add-feature の主要ワークフローが全テンプレートで定義されていること
- **配布自動化**: テンプレートリポジトリへの自動配布が GitHub Actions で動作していること
- **環流可能性**: プロジェクト経験をプラットフォームハーネスコアに蒸留・環流できる仕組みが整っていること

## 機能要件

### コア機能(MVP)

#### Kiro テンプレート（`kiro-template/`）

**ユーザーストーリー**:
Kiro ユーザーとして、Kiro のネイティブ機能（steering・hooks・specs 等）を活かした SDD ハーネスをすぐに使い始めるために、設定済みのテンプレートリポジトリが欲しい

**受け入れ条件**:
- [ ] `.kiro/steering/` 配下に product.md・tech.md・structure.md・process.md のテンプレートが存在する
- [ ] `.kiro/hooks/` 配下に setup-project・add-feature のワークフローが定義されている
- [ ] `docs/` 配下に 6 つの永続ドキュメントテンプレートが存在する（product-requirements・functional-design・architecture・repository-structure・development-guidelines・glossary）
- [ ] `.steering/_template/` 配下に requirements.md・design.md・tasklist.md のテンプレートが存在する
- [ ] README.md にセットアップ手順と Claude Code ハーネスからの移行対応表が記述されている

**優先度**: P0（完成済み）

---

#### Codex CLI テンプレート（`codex-template/`）

**ユーザーストーリー**:
Codex CLI ユーザーとして、AGENTS.md を介した文脈注入と SDD フローを活かしたハーネスをすぐに使い始めるために、設定済みのテンプレートリポジトリが欲しい

**受け入れ条件**:
- [ ] `AGENTS.md` に SDD プロセス原則・プロダクト定義・技術スタック・リポジトリ構造・ワークフローの 5 セクションが存在する
- [ ] `.devcontainer/` に Codex CLI + GitHub CLI がインストールされた開発環境が定義されている
- [ ] `.steering/_template/` 配下に requirements.md・design.md・tasklist.md のテンプレートが存在する
- [ ] `docs/` 配下に 6 つの永続ドキュメントテンプレートが存在する
- [ ] README.md にセットアップ手順と Claude Code ハーネスからの移行対応表が記述されている
- [ ] ONBOARDING.md にセットアップ〜 SDD ワンサイクルまでの手順が記述されている

**優先度**: P0（完成済み）

---

#### テンプレート配布自動化

**ユーザーストーリー**:
ハーネスエンジニアとして、`platform-harness-engineering` で更新したテンプレートを各配布先リポジトリに自動反映するために、GitHub Actions による自動配布の仕組みが欲しい

**受け入れ条件**:
- [ ] `.github/workflows/` に各テンプレートの配布ワークフローが存在する
- [ ] Kiro テンプレートが `platform-harness-for-kiro` リポジトリに自動配布される
- [ ] Codex CLI テンプレートが `platform-harness-for-codex` リポジトリに自動配布される
- [ ] 配布時に GitHub Release が自動作成される

**優先度**: P0（完成済み）

---

#### 概念マッピングドキュメント

**ユーザーストーリー**:
SDD ハーネス経験者として、使用ツールを変更・追加する際のコストを下げるために、ツール間の概念対応表が欲しい

**受け入れ条件**:
- [ ] Claude Code ハーネスの構成要素（CLAUDE.md・skills/・memory/・.steering/・.mcp.json）と各ツール機能の対応表が `docs/` に存在する
- [ ] 各対応において「完全移植」「部分移植」「代替手段あり」「未対応」が明記されている

**優先度**: P1

---

### 将来的な機能(Post-MVP)

#### 新ツール向けテンプレートの追加プロセス

新しい AI ツールが登場・普及したとき、標準プロセスに従って `{tool}-template/` を追加し、publish ワークフローを拡張できるようにする。

テンプレートに含めるべき要素（チェックリスト形式）を定義し、新ツール対応のコストを最小化する。

**優先度**: P2

---

#### 環流プロセスの多ツール対応

各ツールでのプロジェクト経験をプラットフォームハーネスコアに蒸留・環流するプロセスを、ツールを問わず機能するワークフローとして再定義する。

**優先度**: P2

---

#### Kiro カスタムエージェント

Claude Code の `skills/`（steering・add-feature・review-docs 等）に相当する Kiro カスタムエージェントを定義する。

**優先度**: P2

## 非機能要件

### ユーザビリティ
- テンプレート複製から最初のスペック作成開始まで 10 分以内
- 各テンプレートファイルにコメント（`<!-- 例: ... -->`）で記入ガイドが含まれる
- README.md を読むだけでセットアップと SDD 開始の流れが分かる

### 保守性
- ツールごとにディレクトリ（`kiro-template/`・`codex-template/`）が分離されており、変更が局所的に行える
- 新ツール追加時に既存テンプレートに影響を与えない構造になっている
- 配布自動化により、`platform-harness-engineering` の更新が配布先リポジトリに確実に反映される

### 信頼性
- テンプレートファイルはそのままコピーして使えること（未記入のプレースホルダーが明確に区別できる）
- 各ツール固有の構文エラーがないこと

## スコープ外

- 各 AI ツール本体の機能開発・改修（外部ツール）
- 特定プロジェクト向けのカスタマイズ（このリポジトリはテンプレートのみ）
- CI/CD パイプラインの整備（プロジェクト固有のため対象外）
- 配布先リポジトリ（for-kiro・for-codex）の機能開発（それぞれ独立したリポジトリで管理）
