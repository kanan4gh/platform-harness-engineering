# 開発ガイドライン (Development Guidelines)

## このドキュメントの構成

本ガイドラインは2部構成である:

- **第1部**: このテンプレートリポジトリ自体を改善・貢献する際のガイドライン
- **第2部**: このテンプレートを使って開発するプロジェクト向けのガイドライン（プレースホルダー）

# 第1部: ハーネス開発ガイドライン

## 開発フロー

### ブランチ戦略

```
main（テンプレートの安定版）
└── feature/[改善内容]  # 新機能・改善
└── fix/[修正内容]      # バグ修正・誤記修正
```

- `main` へ直接コミットしない（機能実装コードも含む）
- 全変更はフィーチャーブランチ → PR → マージの手順で行う
- 環流（他プロジェクトでの知見を反映する場合）も同様にブランチを切る

### コミットメッセージ規約

Conventional Commits に準拠する:

```
<type>(<scope>): <subject>
```

| type | 用途 |
|------|------|
| `feat` | 新しいテンプレートファイル・機能の追加 |
| `fix` | テンプレートの誤記・不備の修正 |
| `docs` | ドキュメントの更新 |
| `refactor` | テンプレート構造の整理（機能変更なし） |
| `chore` | `.gitignore` 等の設定変更 |

例:
```
feat(steering): process.md に specs 不使用の指針を追加
fix(template): requirements.md の Issue URL 記入欄のフォーマットを修正
docs(architecture): 環境設定戦略セクションを追加
```

### PR プロセス

**作成前チェックリスト**:
- [ ] 変更内容が `docs/functional-design.md` の設計指針に矛盾しないか
- [ ] テンプレートファイルの記入ガイド（`<!-- 例: ... -->`）が含まれているか
- [ ] `docs/repository-structure.md` の構造と一致しているか
- [ ] 環流対象の変更の場合、普遍性・再現性・抽象性の基準を満たすか（`docs/architecture.md` 参照）

**PR テンプレート**:
```markdown
## 概要
[変更内容の簡潔な説明]

## 変更理由
[なぜこの変更が必要か。環流の場合は元プロジェクト・経験を記述]

## 変更内容
- [変更点1]
- [変更点2]

## 動作確認
- [ ] テンプレートを複製して新規プロジェクトで動作確認した
- [ ] steering ファイルが Kiro のコンテキストに正しく反映されることを確認した

## 関連 Issue
Closes #[番号]
```

## テンプレートファイルの品質基準

このリポジトリの「コード品質」はテンプレートファイルの品質を指す。

### Steering ファイルの品質基準

| 観点 | 基準 |
|------|------|
| フロントマター | `inclusion: always` が設定されている |
| 記入ガイド | `<!-- 例: ... -->` 形式のガイドが含まれている |
| 具体性 | プレースホルダーが何を記入すべきか明確である |
| 完結性 | ファイル単体で役割が理解できる |

**良い例**:
```markdown
---
inclusion: always
---

# プロダクト定義

## プロダクト名
<!-- 例: MyApp - ユーザー向けタスク管理サービス -->
[プロダクト名とサブタイトルを記入]

## ビジョン
<!-- 例: エンジニアが CLI から離れずにタスク管理できるツールを提供する -->
[このプロダクトが目指す世界観を 1〜3 文で記入]
```

**悪い例**:
```markdown
# プロダクト定義

ここにプロダクトの情報を書く
```
（何を書けばよいか不明確）

### スペックテンプレートの品質基準

| ファイル | 必須項目 |
|---------|---------|
| `requirements.md` | Issue URL 記入欄・ユーザーストーリー形式・受け入れ条件チェックリスト・スコープ外欄 |
| `design.md` | 実装アプローチ欄・変更対象ファイル欄・技術的判断記入欄 |
| `tasklist.md` | チェックリスト形式・実装→テスト→ドキュメント更新の順序が明示されている |

### フックファイルの品質基準

- フック API が現時点の Kiro バージョンで動作確認済みであること
- フックの目的と動作がコメントまたは README で説明されていること
- kiro-cli を必要とするコマンドを含まないこと

## 環境セットアップ（このリポジトリ自体を開発する場合）

このリポジトリは Markdown / JSON ファイルのみで構成されるため、特別なランタイムや依存関係は不要である。

**必要なもの**:
- Kiro IDE（テンプレートの動作確認用）
- Git（バージョン管理）
- GitHub アカウント（PR・Issue 管理）

**セットアップ手順（GUI のみ）**:
1. GitHub でこのリポジトリをフォークまたはクローン
2. Kiro でフォルダを開く
3. `.kiro/steering/` が Kiro のコンテキストに読み込まれることを確認
4. 変更を加えてフィーチャーブランチへコミット（Kiro の Git 統合 UI を使用）
5. GitHub 上で PR を作成

## リリースフロー

### 通常のリリース（機能追加・修正）

1. フィーチャーブランチを `main` へマージ
2. GitHub Release を作成（`gh release create` または GitHub Web UI）
3. リリースに紐づく Issue をクローズ

### 初回リリース（Bootstrap → リリースフェーズへの移行）

このリポジトリは Claude CLI でブートストラップされた Kiro ハーネスである。初回リリース前に必ず以下を確認する。

**前提**: `docs/architecture.md` の「ブートストラップ戦略」を参照

**初回リリースの条件**: `docs/architecture.md` に定義された受け入れテストの全シナリオが合格していること

**受け入れテスト実施手順**:
1. このリポジトリを "Use this template" で複製し、テスト用の新リポジトリを作成する
2. Kiro でテスト用リポジトリを開く
3. `docs/architecture.md` の受け入れテスト・シナリオ1〜4を順番に実行する
4. 全項目が合格したら初回リリースを行う

**Claude CLI 環境（`.claude/`・`CLAUDE.md`・`skills/`・`memory/`）について**:
- 初回リリース時点では **残したままで公開してよい**
- Kiro ハーネスで自分自身の改善が完結できると判断したタイミングで削除する（自走フェーズ）
- 削除は必須ではない。ブートストラップの文脈を残しておく価値もある

1. GitHub Issue を作成する
